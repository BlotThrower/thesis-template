\chapter{Versuchsurchführung}\label{ch:experiments}
\section{Aufnahme der Testdaten}
Über vier Monate hinweg werden Trainingsdaten von unterschiedlichen Straßentypen gesammelt. Der Fokus liegt auf den häufigsten Arten von Belägen, die in üblichen europäischen Städten anzutreffen sind: 

\begin{itemize}
    \item \textbf{Asphalt}
    \item \textbf{Verbund} (Betonstein- bzw.\ Verbundpflaster)
    \item \textbf{Pflastersteine} (Naturstein- bzw.\ Kopfsteinpflaster)
    \item \textbf{Feldweg} (Unbefestigter Weg, Erde, Schotter)
\end{itemize}

Diese werden zusätzlich in Qualitätsstufen unterteilt, um zu testen, welche maximale Granularität bei der Klassifizierung möglich und brauchbar ist.

\begin{itemize}
    \item \textbf{Asphalt}
    \begin{itemize}
        \item \textbf{Gut:} Glatte Oberfläche, frisch asphaltiert, kaum messbare Erschütterungen.
        \item \textbf{Mittel:} Raue Oberfläche, leichte Risse, ausgebesserte oder geflickte Stellen.
        \item \textbf{Schlecht:} Größere Schäden, Risse, Schlaglöcher, Baumwurzeln, starke und abrupte vertikale Beschleunigungen.
    \end{itemize}
    
    \item \textbf{Verbund}
    \begin{itemize}
        \item \textbf{Gut:} Eben verlegte Steine, enge Fugen, geschlossene, flache Oberfläche.
        \item \textbf{Mittel:} Leichte Setzungen im Untergrund, Fugen sind deutlich beim Fahren spürbar.
        \item \textbf{Schlecht:} Starke Fugen, hochstehende Kanten oder fehlende Steine.
    \end{itemize}
    
    \item \textbf{Pflastersteine}
    \begin{itemize}
        \item \textbf{Gut:} Gerades historisches Pflaster, relativ flach, erzeugt ein konstantes Vibrationsmuster.
        \item \textbf{Mittel:} Kopfsteinpflaster mit intakten Fugen. Erzeugt durchgehend starke Vibrationen.
        \item \textbf{Schlecht:} Sehr grobes, unregelmäßiges Pflaster mit tiefen Fugen, lockeren Steinen oder tiefen Kuhlen. Erzeugt extreme Erschütterungen.
    \end{itemize}
    
    \item \textbf{Feldweg}
    \begin{itemize}
        \item \textbf{Gut:} Festgefahrener, trockener Erdboden oder sehr feiner, verdichteter Schotter.
        \item \textbf{Mittel:} Loser Schotter, leichte Unebenheiten, Waschbrettmuster, flache Pfützen oder Spurrillen.
        \item \textbf{Schlecht:} Grobe Steine, tiefe Matschlöcher, Wege in oder am Rand von Wäldern.
    \end{itemize}
    \item \textbf{Stillstand} (Wenn Geschwindigkeit zu niedrig für erkennbare Datenmuster ist.)
\end{itemize}

Das Grundgerüst für Preprocessing und Datenverarbeitung für die verschiedenen ML-Modelle sind größenteils identisch und unterscheiden sich nur bedingt durch verschiedene Strukturen in den verwendeten Libraries. Deswegen wird hier nur exemplarisch auf eines der Prozesse eingegangen.

Das Projekt ist in Form eines Python-Notebooks aufgebaut. Im Folgenden wird der Aufbau schrittweise erklärt. Um die Reproduzierbarkeit der Ergebnisse sicherzustellen und die verfügbare Hardware effizient zu nutzen, wird das PyTorch-Modell zu Beginn initialisiert:

\begin{lstlisting}[language=Python, caption={Initialisierung der Zufallsgeneratoren und Hardware-Zuweisung für NNs.}, label={lst:setup}]
seed = 42
np.random.seed(seed)
random.seed(seed)
torch.manual_seed(seed)

if torch.cuda.is_available():
    torch.cuda.manual_seed_all(seed)
    torch.backends.cudnn.deterministic = True
    torch.backends.cudnn.benchmark = False
    device = torch.device("cuda")
elif torch.backends.mps.is_available():
    device = torch.device("mps")
else:
    device = torch.device("cpu")

print(f"Device: {device}")
\end{lstlisting}

Der in Listing \ref{lst:setup} dargestellte Codeausschnitt erfüllt drei wesentliche Funktionen für das Training der NNs:
\begin{itemize}
    \item \textbf{Reproduzierbarkeit:} Durch das Festlegen eines Startwerts (\textit{Seed}) für alle verwendeten Zufallsgeneratoren (\texttt{numpy}, \texttt{random} und \texttt{torch}) wird sichergestellt, dass Prozesse wie das Bestimmen der Gewichte bei jedem Durchlauf identisch sind.
    \item \textbf{Determinismus:} Die \texttt{cudnn}-Parameter zwingen die Grafikkarte dazu, nur deterministische Faltungs-Algorithmen zu verwenden. Dies verhindert Abweichungen, die durch dynamische Optimierungen der Hardware entstehen könnten.
    \item \textbf{Hardware-Zuweisung:} Der Code prüft automatisch die Hardware und weist das Modell der vorhandenen bestmöglichen Art zu, um das Training zu beschleunigen. Dabei werden Nvidia-GPUs (\texttt{cuda}) und Apple-Silicon-Chips (\texttt{mps}) unterstützt. Werden diese nicht gefunden, wird auf den Hauptprozessor (\texttt{cpu}) zurückgegriffen.
\end{itemize}

Um die Sensordaten unabhängig von der individuellen Ausrichtung des Smartphones am Fahrrad auswerten zu können wird die vertikale Beschleunigung isoliert:
 
\begin{lstlisting}[language=Python, caption={Berechnung der reinen vertikalen Beschleunigung durch Projektion auf den Schwerevektor.}, label={lst:vertical_acc}]
def calculate_vertical_acceleration(acc_signals, gravity_signals):
    #1
    g_norms = np.linalg.norm(gravity_signals,
    axis=1, keepdims=True)

    #2
    g_norms[g_norms == 0] = 1.0

    #3
    g_hat = gravity_signals / g_norms
    
    #4
    a_v = np.sum(acc_signals * g_hat, axis=1)

    return a_v.reshape(-1, 1)
\end{lstlisting}

Die in Listing \ref{lst:vertical_acc} gezeigte Funktion nutzt die von während der Fahrt aufgenommenen Gravitationsdaten. In Schritt 1 wird die Länge des Gravitationsvektors pro Zeitschritt errechnet. In Schritt 2 wird eine Division durch 0 verhindert. Schritt 3: Ein Numpy-Array an Gravitationsvektoren wird errechnet. Schritt 4 besteht aus einem Skalarprodukt der Accelerometerdaten mit ihren korrespondierenden Gravitationsvektoren ($a_x$ mit $g_x$, $a_y$ mit $g_y$, $a_z$ mit $g_z$). Diese werden aufsummiert, um ein eindimensionales Array mit vertikalen Beschleunigungen zu erhalten.

Durch diese Projektion werden alle Störfaktoren wie Bremsen oder Lenken eliminiert. Nur die vertikalen Erschütterungen des Fahrrads bleiben übrig.

Um bessere Auskunft über die Qualität der Straße unabhängig ihres Typs zu erhalten wird ein approximierter IRI aus den Sensordaten berechnet:

\begin{lstlisting}[language=Python, caption={Berechnung und Interpolation des fahrradbasierten IRI.}, label={lst:calculate_iri}]
def calculate_IRI
    (loc_interp, acc_vert_filtered, acc_time, az_modifier=0.035):
    v = loc_interp[:, 0]
    az = acc_vert_filtered[:, 0]

    moving_mask = v > 0.5
    v_moving = v[moving_mask]
    az_moving = az[moving_mask]
    time_moving = acc_time[moving_mask]

    iri_results = []
    points_per_block = 500

    for i in range(0, len(az_moving) 
    - points_per_block, points_per_block):
        group_az = az_moving[i : i + points_per_block]
        group_v = v_moving[i : i + points_per_block]
        group_time = time_moving[i]

        az_car = group_az * az_modifier
        az_rms = np.sqrt(np.mean(az_car**2))
        mean_v = np.mean(group_v)

        iri_val = (az_rms / mean_v) * 100 if mean_v > 0 else 0

        iri_results.append({
            'Time (s)': group_time,
            'IRI': iri_val,
            'az': az_rms,
            'v': mean_v,
            'len': len(group_az),
        })

    df_iri_final = pd.DataFrame(iri_results)

    iri_interp_func = interp1d(
        df_iri_final['Time (s)'].values,
        df_iri_final['IRI'].values,
        kind='linear',
        bounds_error=False,
        fill_value='extrapolate',
    )

    iri_interp = iri_interp_func(acc_time).reshape(-1, 1)
    iri_interp[~moving_mask] = 0.0

    return df_iri_final, iri_interp
\end{lstlisting}

Die in Listing \ref{lst:calculate_iri} dargestellte Funktion verarbeitet die vertikale Beschleunigung und die Geschwindigkeit in drei Schritten:
\begin{enumerate}
    \item \textbf{Datenfilterung (Masking):} Zunächst werden alle Messpunkte herausgefiltert, bei denen die Geschwindigkeit $v \le 0{,}5$\,m/s beträgt. So wird sichergestellt, dass der errechnete IRI-Wert nicht durch zu niedrige Geschwindigkeit verzerrt wird.
    \item \textbf{Segmentierung und Berechnung:} Das gefilterte Signal wird in Fenster von 500 Messpunkten unterteilt. Für jedes Fenster wird die vertikale Beschleunigung mit einem fahrradspezifischen Faktor (\texttt{az\_modifier}) skaliert und deren quadratischer Mittelwert errechnet. Der lokale IRI ergibt sich anschließend aus dem Verhältnis des RMS-Wertes zur Durchschnittsgeschwindigkeit im jeweiligen Fenster.
    \item \textbf{Interpolation:} Die ML-Modelle benötigen Feature-Vektoren in der ursprünglichen Abtastrate der Acceleratoren (dies wurde als Standard festgelegt, da es die wichtigsten Werte sind). Die Ergebnisse werden linear auf die Zeitachse der Accelerometer-Messreihe interpoliert. Phasen des Stillstands erhalten den IRI-Wert null.
\end{enumerate}
